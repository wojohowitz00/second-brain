# Phase 13.3: Correction Mechanism

**Goal**: Let users re-classify notes that were categorized incorrectly. Selecting a new PARA category updates the database and moves the markdown file on disk.

## User Review Required

> [!IMPORTANT]
> Corrections trigger a **file move** on disk — the note's markdown file is relocated from the old PARA folder to the new one. This is intentionally destructive (the old path ceases to exist).

## Proposed Changes

### 1. Correction View

#### [NEW] `Sources/SecondBrain/UI/CorrectionView.swift`
- Standalone view accessible from the sidebar (new `SidebarItem.corrections` between Dashboard and Search)
- **List of uncorrected notes** (from `AppDatabase.uncorrectedNotes()`)
- Each row shows:
  - Original text (truncated)
  - Current category pill
  - Picker to select new category (`Projects` / `Areas` / `Resources` / `Archives`)
  - "Apply" button to commit the correction
- After applying, the note disappears from the list with animation

### 2. Correction Service

#### [NEW] `Sources/SecondBrain/Services/CorrectionService.swift`
- `func applyCorrection(note: Note, newCategory: String, vaultPath: String, database: AppDatabase) throws`
- Steps:
  1. Compute old file path: `vaultPath/note.destinationPath`
  2. Compute new destination: `newCategory_folder/note.classifiedFilename`
  3. Move file via `FileManager.default.moveItem`
  4. Call `note.applyCorrection(newCategory:)` and update in DB
  5. Update `note.destinationPath` to the new location
- Handles missing source file gracefully (still updates DB)
